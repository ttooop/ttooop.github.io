Kafka

一、介绍

背景：

事件流在各种行业场景中有广泛的应用，例如实时处理支付和金融交易； 实时跟踪和监控汽车、卡车、车队和货运； 持续捕获和分析来自物联网设备或其他设备的传感器数据； 收集并立即响应客户互动和订单； 监测住院病人，预测病情变化，确保在紧急情况下及时治疗；连接、存储和提供公司不同部门产生的数据。 作为数据平台、事件驱动架构和微服务的基础。

1.1、功能与特性

Kafka作为事件流处理平台，主要功能有：

  1. 发布和订阅事件流——>publish&subscribe
  2. 持久可靠地存储事件信息——>store
  3. 实时或后置处理数据——process

Kafka在实现以上功能时，具有强大的特性，主要表现在：

  • distributed
  • hilghly scalable
  • elastic
  • Fault-tolerant
  • secure

1.2、概念介绍

  1. 一个事件经常包含三部分信息：key value timestamp
  2. producers生产事件信息到Kafka中
  3. consumers订阅事件，从Kafka消费事件信息
  4. producers和consumers是解耦且互不可知的
  5. 事件消息被组织和持久存储在topics中，Kafka通过topics来隔离不同类型的事件消息，一个topic通常对应多个生产者和多个消费者。topic中的事件消息可以在需要时被读取消费，和传统消息系统消费后删除的策略不同，在Kafka可以通过每个主题的配置来定义事件消息保留多久后会被丢弃。
  6. topics是分区的，数据的这种分布式放置对于可伸缩性非常重要，因为它允许客户端应用程序同时从多个代理读取和写入数据。当一个新事件发布到一个主题时，它实际上是附加到主题的分区之一。具有相同事件键（例如，客户或车辆 ID）的事件被写入同一个分区，并且 Kafka 保证给定主题分区的任何消费者将始终以与写入事件完全相同的顺序读取该分区的事件。
  7. topic是有复制备份的，来保证高可用和容错性。

二、设计

2.1、目标

  1. 高吞吐：以支持大体量事件流
  2. 需要能够支持大量数据积压，以支持来自离线系统的定期数据加载
  3. 支持低延迟交付
  4. 分布式、实时性、分区可扩展
  5. 支持容错、高可用

2.2、持久化

Kafka消息的存储和缓存依赖于文件系统，but基于磁盘访问慢的背景，借助cache来弥补性能差异

Kafka设计：相比于维护尽可能多的 in-memory cache，并且在空间不足的时候匆忙将数据 flush 到文件系统，我们把这个过程倒过来。所有数据一开始就被写入到文件系统的持久化日志中，而不用在 cache 空间不足的时候 flush 到磁盘。实际上，这表明数据被转移到了内核的 pagecache 中。
