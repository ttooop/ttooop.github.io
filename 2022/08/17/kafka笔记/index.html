一、介绍

背景：

事件流在各种行业场景中有广泛的应用，例如实时处理支付和金融交易； 实时跟踪和监控汽车、卡车、车队和货运； 持续捕获和分析来自物联网设备或其他设备的传感器数据； 收集并立即响应客户互动和订单； 监测住院病人，预测病情变化，确保在紧急情况下及时治疗；连接、存储和提供公司不同部门产生的数据。 作为数据平台、事件驱动架构和微服务的基础。

1.1、功能与特性

Kafka作为事件流处理平台，主要功能有：

  1. 发布和订阅事件流——>publish&subscribe
  2. 持久可靠地存储事件信息——>store
  3. 实时或后置处理数据——process

Kafka在实现以上功能时，具有强大的特性，主要表现在：

  • distributed
  • hilghly scalable
  • elastic
  • Fault-tolerant
  • secure

1.2、概念介绍

  1. 一个事件经常包含三部分信息：key value timestamp
  2. producers生产事件信息到Kafka中
  3. consumers订阅事件，从Kafka消费事件信息
  4. producers和consumers是解耦且互不可知的
  5. 事件消息被组织和持久存储在topics中，Kafka通过topics来隔离不同类型的事件消息，一个topic通常对应多个生产者和多个消费者。topic中的事件消息可以在需要时被读取消费，和传统消息系统消费后删除的策略不同，在Kafka可以通过每个主题的配置来定义事件消息保留多久后会被丢弃。
  6. topics是分区的，数据的这种分布式放置对于可伸缩性非常重要，因为它允许客户端应用程序同时从多个代理读取和写入数据。当一个新事件发布到一个主题时，它实际上是附加到主题的分区之一。具有相同事件键（例如，客户或车辆 ID）的事件被写入同一个分区，并且 Kafka 保证给定主题分区的任何消费者将始终以与写入事件完全相同的顺序读取该分区的事件。
  7. topic是有复制备份的，来保证高可用和容错性。

二、设计

2.1、目标

  1. 高吞吐：以支持大体量事件流
  2. 需要能够支持大量数据积压，以支持来自离线系统的定期数据加载
  3. 支持低延迟交付
  4. 分布式、实时性、分区可扩展
  5. 支持容错、高可用

2.2、持久化

Kafka消息的存储和缓存依赖于文件系统，但是磁盘访问每次IO需要经过寻道 旋转 数据传输三个步骤，访问慢的背景，Kafka通过两种方式来弥补性能差异

  1. 顺序追加写：
    • Kafka采用顺序写的方式提高磁盘的写入性能
    • Kafka中每个partition是有序的、不可变的消息序列，新的消息不断追加到partition末尾，在存储上，Kafka将partition划分为多个segment，每个segment对应一个物理文件，对segment文件追加写
    • Kafka文件结构：每个partition日志在物理上按照大小被分成多个segment，包括index file和data file，一一对应，成对出现，后缀”.index”和“.log”分别表示为 segment 索引文件、数据文件。segment 文件名为上一个 segment 文件最后一条消息的 offset 值
  2. 借助page cache弥补性能差异
    • Kafka设计：相比于维护尽可能多的 in-memory cache，并且在空间不足的时候匆忙将数据 flush 到文件系统，我们把这个过程倒过来。所有数据一开始就被写入到文件系统的持久化日志中，而不用在 cache 空间不足的时候 flush 到磁盘。实际上，这表明数据被转移到了内核的 pagecache 中。
    • page cache缓存的是最近会被使用到的磁盘数据，利用的是「时间局部性」原理。而预读到page cache中的磁盘数据，又利用了「空间局部性」原理。Kafka作为消费队列，消息先顺序写入，又立马会被消费者读取到，无疑是非常契合上述两条局部性原理
    • producer生产消息到broker时，broker会使用pwrite()系统调用按照偏移量写入数据，此时数据会先写入page cache
    • consumer消费消息时，broker使用sendfile()系统调用，零拷贝地将数据从page cache传输到broker的socket buffer，再通过玩过传输。
    • page cache中的数据会随着内核中 flusher 线程的调度以及对 sync()/fsync() 的调用写回到磁盘，就算进程崩溃，也不用担心数据丢失。另外，如果 consumer 要消费的消息不在page cache里，才会去磁盘读取，并且会顺便预读出一些相邻的块放入 page cache，以方便下一次读取。
    • 因此如果 Kafka producer 的生产速率与 consumer 的消费速率相差不大，那么就能几乎只靠对 broker page cache 的读写完成整个生产 - 消费过程，磁盘访问非常少。

2.3、效率

两种可能降低效率的场景：

  • 大量的IO操作
  • 过多的字节copy

针对上面两个问题，Kafka通过批量压缩和零拷贝来优化

  1. 批量与压缩
    • 批量发送：通过batch.size linger.ms来配置批量发送条件，当达到定义的批次大小或达到定义的延迟时间后，就进行批量的发送
    • 压缩算法：lz4、snappy、gzip。Kafka 2.1.0 正式支持 ZStandard，【端到端的压缩设计】Producer、Broker 和 Consumer 使用相同的压缩算法，在 producer 向 Broker 写入数据，Consumer 向 Broker 读取数据时甚至可以不用解压缩，最终在 Consumer Poll 到消息时才解压，这样节省了大量的网络带宽和磁盘开销。
  2. 零拷贝
    • 数据使用时，只会被复制一份到page cache中，节省每次拷贝到用户空间内存中，再从用户空间进行读取的消耗

2.4、生产者

  • 生产者将消息直接生产到主分区的服务器上，client控制消息发送到哪个分区

2.5、消费者

从消费者角度来看，需要满足两个基本诉求：

  • 同一个topic可以被多个消费者消费，且一条消息可以被消费多次
  • 当消费者本身是集群（消费组）时，每一个消息只能分发给集群中的一个消费者进行处理
  1. push vs pull
    • Kafka采用的pull拉模式，consumer可以根据自身负载情况主动拉消息，并且可以自由聚合批量数据
  2. consumer position
    • Kafka topic由有序的partition组成，每个partition在任意时间内只能被订阅了这个topic的consumer group中的一个consume消费
    • 所以在partition中以offset标记消费的位置来表示消费状态，Kafka利用二分法来查找对应offset的消息位置。
    • consumer可以通过offset设置回退到之前的offset再次消费之前的数据
  3. 稀疏索引
    • 为了加快读操作，如果只需要在内存中维护一个「从 offset 到日志文件偏移量」的映射关系即可，每次根据 offset 查找消息时，从哈希表中得到偏移量，再去读文件即可
    • 将消息划分成若干个 block，只索引每个 block 第一条消息的 offset 即可，先根据大小关系找到 block，然后在 block 中顺序搜索，这便是 Kafka “稀疏索引” 的设计思想。
    • 有了稀疏索引，当给定一个 offset 时，Kafka 采用的是二分查找来高效定位不大于 offset 的物理位移，然后找到目标消息。

2.6、消息交付语义

消息交付语义涉及两个问题：

  • 发布消息的持久性保证
  • 消费消息的保证

三种交付语义：

  1. At most once：至多一次，表示消息可能会丢失，且不会重传
  2. At least once：至少一次，表示会有重传机制，消息一定不会丢失，但是可能会有重复消息的情况
  3. Exactly once：有且只有一次，即消息生产和消费只被传递一次

  • Kafka默认的级别是At least once，会有重传机制，会先读消息，处理消息，然后提交保存offset，如果提交保存offset失败可能存在重复消费的问题。

  • Kafka可以通过禁用重传+处理消息之前提交offset的形式来实现at most once，可能存在消息丢失的问题

  • exactly once级别通过consumer和offset两阶段提交的方式来保证

Q：如何保证消息不重复消费？

A：消息重复消费的根本原因是已经消费成功的consumer提交保存offset失败

解决方案有

1. 消费消息服务做幂等校验
1. 关闭自动提交，开发者在代码中手动提交offset

2.7、多副本

备份单位：topic partition

一主多从结构：一个leader和多个followers，leader负责读写请求，follower副本负责和leader同步消息，当leader出现故障时，follower才会有机会被选举成新的leader副本并且对外提供服务。

通常，partitions比broker多，并且leaders在brokers之间均匀分布。

Kafka动态维护了一个同步状态的备份集合ISR（in-sync replicas），该集合中的节点和lederhosen保持高度一致，ISR：

  • 只有ISR集合中的节点可以选为leader
  • 一条消息必须被ISR集合中所有节点读取并追加到日志中，才视为完成提交

在多副本机制下，可用性和持久性的保证通过acks来确认：

  1. acks=0：不等待broker确认直接返回
  2. acks=1：leader保存成功就直接返回
  3. acks=-1/all：所有备份保存成功后返回

多副本管理：

​ 选择一个broker作为controller节点，来检测broker的级别故障
